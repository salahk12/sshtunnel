# SSH Tunnel Panel

پنل وب سبک (نوشته‌شده با Go) برای ساخت و مدیریت تانل‌های SSH بین دو سرور — جایگزین مدرنِ
اسکریپت دستیِ `ssh -L` + `systemd`. هر تانل یک سرویس systemd مستقل است، ولی به‌جای
فراخوانی `/usr/bin/ssh`، خودِ باینریِ پنل اتصال SSH و port-forward را برقرار می‌کند تا
**تعداد worker** و **آمار ترافیک ورودی/خروجی** هم پشتیبانی شود.

## قابلیت‌ها
- 🔐 پنل وب با **یوزرنیم/پسورد** و **web path** قابل تنظیم
- ➕ افزودن تانل با **آیپی + یوزرنیم + پسورد** (کلید به‌صورت خودکار مثل `ssh-copy-id` روی سرور خارج نصب می‌شود) یا با **آیپی + یوزرنیم + کلید SSH موجود**
- ⚙️ انتخاب **نوع رمزنگاری** (aes128-gcm, aes256-gcm, chacha20-poly1305, …)
- 🧵 انتخاب **تعداد worker** (چند اتصال موازی، توزیع round-robin)
- 📊 مشاهده‌ی زنده‌ی **ترافیک ورودی/خروجی** + نمودار + اتصالات فعال + لاگ سرویس
- 🔁 چند **port-forward** در هر تانل، reconnect خودکار، keepalive، فشرده‌سازی
- ↔️ دو حالت تانل: **Local** (ایران به خارج وصل می‌شود) و **Reverse** (خارج به ایران وصل می‌شود؛ ایران هیچ اتصال خروجی نمی‌زند)
- 🎛️ **تیونینگ TCP/SSH** هر تانل: اندازه‌ی بافر کپی، بافر سوکت (SO_SNDBUF/RCVBUF)، TCP_NODELAY، و MSS clamp
- 🚀 هر تانل = یک سرویس **systemd** مستقل (در `systemctl` دیده می‌شود)
- 📦 یک باینری استاتیک، مصرف کم CPU/RAM، نصب با **یک دستور**

## نصب سریع (مثل 3x-ui)
> پیش از انتشار، در `scripts/install.sh` مقدار `REPO` را به ریپوی گیت‌هاب خودتان تغییر دهید.

```bash
bash <(curl -fsSL https://raw.githubusercontent.com/salahk12/sshtunnel/main/scripts/install.sh)
```

نصب‌کننده باینری مناسب معماری سرور (amd64/arm64) را از GitHub Release می‌گیرد، سرویس
`sshtunnel-panel` را می‌سازد و راه می‌اندازد، و در اولین نصب **یوزرنیم/پسورد و web path تصادفی**
تولید و چاپ می‌کند.

تنظیم پورت دلخواه: `PORT=8443 bash install.sh` — حذف کامل: `bash install.sh uninstall`

## اجرا روی همان سرور ایران
این پنل را روی **سرور ایران** نصب کنید (همان‌جایی که قبلاً سرویس `ssh-forward` را می‌ساختید).
سرور خارج فقط باید SSH داشته باشد.

## حالت‌های تانل: Local و Reverse
- **Local (پیش‌فرض):** سرور ایران به خارج وصل می‌شود، پورت را روی ایران bind می‌کند و ترافیک را به خارج می‌فرستد. فورواردر درون‌برنامه‌ای (worker + آمار ترافیک + تیونینگ).
- **Reverse:** برای وقتی که **اتصال خروجی ایران→خارج فیلتر/کند است ولی ورودی خارج→ایران باز است**. اتصال را **سرور خارج** برقرار می‌کند:
  - پنل (روی ایران) با SSH به سرور خارج، یک سرویس `ssh -R` نصب می‌کند که به ایران برمی‌گردد.
  - سرور ایران فقط `sshd` دارد (پنل خودکار `GatewayPorts` را با یک drop-in فعال می‌کند) و پورت عمومی را bind می‌کند.
  - ترافیک: کاربر → `ایران:پورت` → کانال ریورس → خارج → مقصد. **ایران هیچ اتصال خروجی در زمان کار نمی‌زند.**
  - آمار/وضعیت ریورس از سمت ایران به‌صورت محلی خوانده می‌شود (bind پورت با `ss` + شمارش بایت با iptables). در ریورس v1، worker و فشرده‌سازی/سایفر از طریق فلگ‌های `ssh` پشتیبانی می‌شوند.
  - فیلدهای لازم: آی‌پی عمومی سرور ایران، پورت/یوزر SSH ایران، و کرِدیت سرور خارج (برای نصب کانکتور).

## معماری چند-سروره (مرکزی + نودها)
برای مدیریت چند سرور ایران از یک داشبورد واحد:

```
        سرور مرکزی (Master)  ← داشبورد همه‌ی نودها
              │   │   │   (فقط ترافیک کنترلی JSON)
        ┌─────┘   │   └─────┐
      نود ۱      نود ۲      نود ۳   ← سرورهای ایران (تانل‌ها به‌صورت systemd مستقل)
        │          │          │
     سرور خارج  سرور خارج  سرور خارج
```

- **هر نصب یک «نود» است** و تانل‌هایش را مستقل با systemd اجرا می‌کند.
- روی یکی از سرورها **حالت Master** را روشن کنید (تنظیمات → «این سرور مرکزی است» یا `MASTER=1 bash install.sh`).
  آن سرور یک تب **«نودها»** می‌گیرد و سرور محلی هم خودکار به‌عنوان نود ثبت می‌شود.
- **افزودن نود:** در تب «نودها» روی *افزودن نود* بزنید و آدرس پنل نود + توکن آن را وارد کنید.
  توکن هر نود را با `sshtunnel-panel node-token` یا از خروجی نصب بگیرید.
- **مدل ارتباط:** Master → Node روی HTTPS/HTTP با bearer token. سرور مرکزی به API هر نود وصل می‌شود؛
  پس باید پورت پنل هر نود به‌روی Master باز باشد (برای امنیت بیشتر، فقط IP مرکزی را در فایروال مجاز کنید).

### مقاومت در برابر خرابی (مهم)
تانل‌ها روی **خودِ نود** به‌صورت سرویس‌های مستقل systemd اجرا می‌شوند. بنابراین:
- اگر **سرور مرکزی** خاموش شود → هیچ تانلی قطع نمی‌شود؛ نودها مستقل کار می‌کنند و وقتی Master برگردد دوباره وضعیت را می‌خواند.
- اگر یک **نود** از دسترس خارج شود → فقط همان نود در داشبورد «آفلاین» می‌شود؛ بقیه و خودِ Master سالم می‌مانند.

## دستورات CLI
```bash
sshtunnel-panel serve                 # اجرای پنل (سرویس systemd همین را صدا می‌زند)
sshtunnel-panel admin --show          # نمایش آدرس و یوزرنیم
sshtunnel-panel admin --password XXX  # تغییر رمز از خط فرمان
sshtunnel-panel admin --random-path   # تولید web path تصادفی جدید
sshtunnel-panel admin --master on     # فعال‌کردن حالت سرور مرکزی
sshtunnel-panel node-token            # نمایش توکن این نود (برای ثبت در Master)
sshtunnel-panel node-token --regenerate   # ساخت توکن جدید (توکن قبلی باطل می‌شود)
sshtunnel-panel run-tunnel --id <ID>  # اجرای یک تانل (داخلی؛ توسط systemd صدا زده می‌شود)
sshtunnel-panel version
```

## توسعه‌ی محلی
```bash
make dev      # اجرا با دایرکتوری‌های محلی dev-config / dev-data (بدون دست‌زدن به /etc)
make build    # ساخت باینری
make cross    # ساخت amd64 + arm64 در dist/
```

## ساختار پروژه
```
main.go                 subcommands: serve / run-tunnel / admin / version
internal/config         پیکربندی پنل (listen, base_path, data_dir, …)
internal/store          ذخیره JSON (کاربر، تانل‌ها)
internal/sshx           کلاینت SSH، تولید/نصب کلید، پین host key
internal/tunnel         مدیریت systemd + فورواردرِ درون‌برنامه‌ای + شمارش ترافیک
internal/web            وب‌سرور، احراز هویت، API، فرانت‌اند embed شده
```

## امنیت
- رمز پنل با bcrypt ذخیره می‌شود؛ کلیدها در `/var/lib/sshtunnel-panel` با دسترسی 0600.
- host key سرور خارج در اولین اتصال پین (TOFU) و در اتصال‌های بعدی بررسی می‌شود.
- توصیه: یک `web path` بلند و تصادفی بگذارید و در صورت امکان پشت TLS قرار دهید
  (مقادیر `tls_cert`/`tls_key` در `config.json`).
